## Front Matter 작성

### 필수 항목

모든 마크다운 파일 상단에 메타데이터를 작성합니다:

```yaml
---
title: "실습 제목"
week: 2
session: 1
awsServices:
  - AWS IAM
  - Amazon S3
learningObjectives:
  - 학습 목표 1
  - 학습 목표 2
prerequisites:
  - 사전 요구사항 1
  - 사전 요구사항 2
---
```

### 항목 설명

| 항목 | 필수 | 설명 | 예시 |
|------|------|------|------|
| `title` | ✅ | 실습 제목 | "IAM 정책 생성 및 관리" |
| `week` | ✅ | 주차 번호 | 2 |
| `session` | ✅ | 세션 번호 | 1 |
| `awsServices` | ✅ | 사용하는 AWS 서비스 목록 | - AWS IAM<br>- Amazon S3 |
| `learningObjectives` | ✅ | 학습 목표 목록 | - IAM 정책을 생성할 수 있습니다 |
| `prerequisites` | ⭕ | 사전 요구사항 (선택) | - AWS 계정 보유 |

## 마크다운 문법

### 1. AWS 버튼 (3가지 스타일)

#### Primary 버튼 (오렌지색) - 주요 액션
**문법**: `[[버튼명]]`

```markdown
1. [[Create bucket]] 버튼을 클릭합니다
2. [[Next]] 버튼을 클릭합니다
3. [[Save changes]] 버튼을 클릭합니다
```

**사용 예**: Create, Save, Next, Launch 등

---

#### Normal 버튼 (흰색) - 보조 액션
**문법**: `{{버튼명}}`

```markdown
1. {{Upload}} 버튼을 클릭합니다
2. {{Edit}} 버튼을 클릭합니다
3. {{Refresh}} 버튼을 클릭합니다
```

**사용 예**: Upload, Edit, Refresh, Delete 등

---

#### Link 버튼 (회색 텍스트) - 취소/뒤로가기
**문법**: `((버튼명))`

```markdown
1. ((Cancel)) 버튼을 클릭합니다
2. ((Back)) 버튼을 클릭합니다
3. ((Close)) 버튼을 클릭합니다
```

**사용 예**: Cancel, Back, Close 등

---

### 2. 필드명/메뉴명 강조

**문법**: `**텍스트**`

```markdown
1. **Bucket name**에 값을 입력합니다
2. **Region**을 선택합니다
3. 왼쪽 메뉴에서 **Policies**를 선택합니다
```

2. Region은 `ap-northeast-2`를 선택합니다

### 5. Alert 박스

#### 공통 Alert (실습 + 데모)

##### [!WARNING] - 경고 (노란색) ⚠️
**사용 시기**: 비용 정보, 리소스 삭제 경고, 데이터 손실 위험

```markdown
> [!WARNING]
> 이 실습에서 생성하는 리소스는 실습 종료 후 **반드시 삭제해야 합니다**.
> RDS 인스턴스(db.t3.micro)는 시간당 약 $0.017이 부과됩니다 (ap-northeast-2 리전 기준).
```

##### [!NOTE] - 참고 (파란색) ℹ️
**사용 시기**: 대기 시간 안내, 추가 설명, 중요하지만 필수는 아닌 정보

```markdown
> [!NOTE]
> 스택 생성에 2-3분이 소요됩니다. **Events** 탭에서 생성 과정을 관찰하세요.
```

##### [!IMPORTANT] - 중요 (보라색) ❗
**사용 시기**: 건너뛰면 실패하는 단계, 필수 확인 사항

```markdown
> [!IMPORTANT]
> 보안 그룹을 올바르게 설정하지 않으면 Redis 클러스터에 연결할 수 없습니다.
```

##### [!TIP] - 팁 (초록색) 💡
**사용 시기**: 모범 사례, 효율적인 방법, 유용한 권장사항

```markdown
> [!TIP]
> 버킷 이름은 전 세계적으로 고유해야 합니다. 계정 ID를 포함하면 충돌을 방지할 수 있습니다.
```

##### [!OUTPUT] - 예상 출력 (파란색) 📋
**사용 시기**: 명령어 실행 결과, 예상되는 화면 출력

```markdown
> [!OUTPUT]
> ```
> PONG
> ```
```

##### [!TROUBLESHOOTING] - 문제 해결 (주황색) 🔧
**사용 시기**: 자주 발생하는 오류, 문제 해결 방법

```markdown
> [!TROUBLESHOOTING]
> **문제**: "Access Denied" 오류가 발생합니다
> **해결**: IAM 역할에 S3 읽기 권한이 있는지 확인하세요.
```

##### [!DOWNLOAD] - 파일 다운로드 (파란색) 📦
**사용 시기**: 실습 파일 제공 (실습 가이드만)

```markdown
> [!DOWNLOAD]
> [week5-1-rds-lab.zip](/files/week5/week5-1-rds-lab.zip)
> - `init_database.sql` - 데이터베이스 초기화 스크립트
> - `README.md` - 실습 가이드
```

---

#### 데모 전용 Alert

##### [!CONCEPT] - 개념 설명 (파란색 카드) 📚
**사용 시기**: 기술 개념 설명, 아키텍처 구성 요소 설명 (데모 가이드만)

```markdown
> [!CONCEPT] CloudFront 오리진 (Origin)
> 오리진은 원본 콘텐츠가 저장된 위치입니다. S3, EC2, ALB 또는 커스텀 HTTP 서버를 사용할 수 있습니다.
> CloudFront는 캐시 미스가 발생하면 오리진에서 콘텐츠를 가져옵니다.
```

**중요**: `[!CONCEPT]`는 데모 가이드에서만 사용하세요. 실습 가이드에서는 `[!NOTE]`나 `[!TIP]`를 사용하세요.

---

#### Alert 사용 가이드라인

**실습 가이드 권장 빈도**:
- `[!WARNING]`: 시작 + 리소스 정리 (필수)
- `[!NOTE]`: 태스크당 1-3개
- `[!TIP]`: 전체 2-5개
- `[!IMPORTANT]`: 전체 1-3개
- `[!DOWNLOAD]`: 파일 제공 시
- `[!TROUBLESHOOTING]`: 전체 1-2개

**데모 가이드 권장 빈도**:
- `[!WARNING]`: 시작 + 리소스 정리 (필수)
- `[!CONCEPT]`: 태스크당 1-3개
- `[!NOTE]`: 태스크당 1-2개
- `[!TIP]`: 전체 2-4개

**피해야 할 사항**:
- ❌ 한 태스크에 Alert 과다 사용 (3개 이하 권장)
- ❌ 실습 가이드에서 `[!CONCEPT]` 사용 (데모 전용)
- ❌ 중요하지 않은 내용에 `[!IMPORTANT]` 사용
- ❌ 예상 출력을 이모지로 표시 (반드시 `[!OUTPUT]` 사용)

1. AWS Management Console에 로그인합니다
2. 상단 검색창에서 `S3`를 검색하고 선택합니다

> [!NOTE]
> 버킷 이름은 전 세계적으로 고유해야 합니다.

✅ **태스크 완료**: S3 버킷이 생성되었습니다.

2. [[Upload]] 버튼을 클릭합니다

## 실습 가이드 vs 데모 가이드

### 차이점 이해하기

| 구분 | 실습 가이드 | 데모 가이드 |
|------|------------|------------|
| **목적** | 학생이 직접 실습 | 교수가 시연 또는 개념 학습 |
| **파일 제공** | ✅ ZIP 파일 제공 | ❌ 파일 제공 안 함 |
| **개념 설명** | 최소화 (실습 집중) | ✅ `[!CONCEPT]` 적극 활용 |
| **단계 상세도** | 매우 상세 (클릭 단위) | 보통 (개념 이해 중심) |
| **제목 예시** | "RDS Multi-AZ 배포 및 페일오버 확인" | "CloudFront CDN 배포 (데모)" |

### 실습 가이드 작성 시

## 자주 묻는 질문

### Q1. 마크다운 파일을 수정하면 바로 반영되나요?
**A**: Git Push 후 자동 배포되면 웹사이트에 반영됩니다.

> [!NOTE]
> 배포 플랫폼이 결정되면 구체적인 배포 시간과 방법이 업데이트됩니다.

**반영 과정**:
1. 마크다운 파일 수정 및 저장
2. Git으로 커밋 및 푸시
3. 자동 배포 시작 (배포 플랫폼 결정 후 시간 명시 예정)
4. 웹사이트에 변경사항 반영

**즉시 확인 방법** (선택사항):
- Kiro에서 마크다운 미리보기: `Cmd+Shift+V` (Mac) 또는 `Ctrl+Shift+V` (Windows)
- 로컬 개발 서버 실행 (개발팀이 설정한 경우만):
  ```bash
  npm run dev
  # http://localhost:3000 에서 실시간 미리보기
  ```

> [!TIP]
> 교수는 보통 로컬 개발 서버를 실행할 필요가 없습니다. Git Push 후 배포된 웹사이트에서 확인하세요.

### Q3. 실습 파일(zip, yaml 등)은 어떻게 업로드하나요?
**A**: `public/files/` 폴더에 파일을 추가하면 자동으로 제공됩니다.

**파일 업로드 단계**:
1. `public/files/week{주차}/` 폴더 생성 (없는 경우)
2. 실습 파일을 해당 폴더에 복사
3. 마크다운에서 파일 경로 참조:
   ```markdown
   > [!DOWNLOAD]
   > [week2-1-iam-policies.zip](/files/week2/week2-1-iam-policies.zip)
   ```

**파일 명명 규칙**:
- 형식: `week{주차}-{세션}-{설명}.{확장자}`
- 예시: `week2-1-iam-policies.zip`

### Q4. AWS 버튼 문법을 잘못 사용하면 어떻게 되나요?
**A**: 문법이 잘못되면 일반 텍스트로 표시됩니다.
- `[[Create bucket]]` ✅ → 오렌지색 버튼
- `[Create bucket]` ❌ → 일반 텍스트

### Q5. 코드 블록에서 언어를 지정하지 않으면?
**A**: 언어를 지정하지 않으면 색상 구분이 적용되지 않습니다. 가능한 언어를 명시하세요.

### Q6. Front Matter에서 필수 항목을 빠뜨리면?
**A**: 오류가 발생하고 페이지가 제대로 표시되지 않습니다. `title`, `week`, `session`은 반드시 포함해야 합니다.

### Q7. 단계 번호가 자동으로 매겨지나요?
**A**: 네! 각 태스크마다 1번부터 시작해도 화면에는 자동으로 연속된 번호가 표시됩니다.

```markdown
## 태스크 1: 버킷 생성
1. 첫 번째 단계
2. 두 번째 단계

## 태스크 2: 파일 업로드
1. 첫 번째 단계  ← 화면에는 "3."로 표시됨
2. 두 번째 단계  ← 화면에는 "4."로 표시됨
```

### Q9. 변경 사항을 어떻게 저장하고 공유하나요?
**A**: Git을 사용하여 버전 관리를 합니다.

**기본 Git 워크플로우**:
```bash
# 1. 변경 사항 확인
git status

# 2. 변경된 파일 추가
git add public/content/week2/2-1-iam-policy.md

# 3. 커밋 메시지 작성
git commit -m "Week 2-1 IAM 정책 실습 가이드 업데이트"

# 4. 원격 저장소에 푸시
git push origin main
```

### Q10. 여러 교수가 동시에 작업할 수 있나요?
**A**: 네, Git 브랜치를 사용하여 협업할 수 있습니다.

**협업 워크플로우**:
```bash
# 1. 새 브랜치 생성
git checkout -b week2-update

# 2. 작업 후 커밋
git add .
git commit -m "Week 2 실습 가이드 수정"

# 3. 브랜치 푸시
git push origin week2-update

# 4. Pull Request 생성하여 리뷰 요청
```

### Q11. 실습 가이드와 데모 가이드 중 어떤 것을 작성해야 하나요?
**A**: 학습 목적에 따라 선택하세요.

**실습 가이드 선택 기준**:
- ✅ 학생이 직접 AWS 리소스를 생성하고 테스트
- ✅ 실습 파일(템플릿, 스크립트 등) 제공 필요
- ✅ 단계별로 매우 상세한 지침 필요
- ✅ 예: "RDS 인스턴스 생성 및 연결"

**데모 가이드 선택 기준**:
- ✅ 교수가 시연하거나 개념 설명 중심
- ✅ 비용이 많이 들거나 시간이 오래 걸리는 작업
- ✅ 개념 이해가 주 목적
- ✅ 예: "CloudFront CDN 동작 원리 이해"

### Q13. 실습 파일을 수정하려면 어떻게 하나요?
**A**: ZIP 파일을 압축 해제하고 수정한 후 다시 압축하세요.

**단계**:
1. 기존 ZIP 파일 다운로드 (예: `week2-1-files.zip`)
2. 압축 해제
3. 파일 수정 (예: `template.yaml` 수정)
4. 다시 ZIP으로 압축
5. `public/files/week2/` 폴더에 업로드 (기존 파일 덮어쓰기)
6. Git으로 변경사항 커밋 및 푸시

**A**: 네, 시스템에 진도 추적 기능이 내장되어 있습니다.

#### 📸 스크린샷 추가 방법

**1단계: 스크린샷 캡처**

**캡처 도구**:
- **macOS**: `Cmd + Shift + 4` (영역 선택)
- **Windows**: `Win + Shift + S` (캡처 도구)
- **추천 도구**: Snagit, Greenshot (무료)

**캡처 가이드라인**:
- ✅ **필요한 부분만 캡처**: 전체 화면 대신 관련 UI만
- ✅ **중요 부분 강조**: 빨간 박스나 화살표로 표시
- ✅ **개인정보 제거**: 계정 ID, 이메일 등 모자이크 처리
- ✅ **일관된 브라우저**: Chrome 또는 Firefox 사용
- ✅ **확대/축소**: 100% 배율 유지
- ✅ **고해상도**: 선명한 이미지 (최소 1280px 너비)

**2단계: 이미지 최적화**

```bash
# 이미지 압축 (TinyPNG 또는 ImageOptim 사용)
# 목표: 500KB 이하, PNG 형식
```

**권장 사양**:
- **형식**: PNG (스크린샷), JPG (사진)
- **크기**: 최대 1920x1080px
- **파일 크기**: 500KB 이하
- **해상도**: 72-96 DPI

**3단계: 파일 저장**

```
public/images/week{주차}/{주차}-{세션}-step{단계}-{설명}.png
```

**파일명 예시**:
- `1-1-step1-s3-search.png` (1주차 1세션 1단계 - S3 검색)
- `2-1-step3-bucket-name.png` (2주차 1세션 3단계 - 버킷 이름)
- `3-2-step5-security-group.png` (3주차 2세션 5단계 - 보안 그룹)

**4단계: 마크다운에 삽입**

```markdown

## 리소스 태그 관리

### 리소스 태그 시스템 개요

이 프로젝트는 **AWS Tag Editor 기반 리소스 관리 시스템**을 사용합니다.

**목적**:
- Week 1-1에서 Tag Editor 실습을 통해 학생들이 리소스 태그 개념 학습
- 이후 모든 실습에서 태그를 활용하여 리소스 추적 및 정리
- 불필요한 비용 발생 방지

**핵심 원칙**:
- 모든 수동 생성 리소스에 표준 태그 3개 추가
- Tag Editor로 실습에서 생성한 모든 리소스를 한 번에 찾기
- CloudFormation 스택 태그 자동 전파 활용

### 표준 태그 3개

모든 실습 리소스에 다음 3개 태그를 필수로 추가합니다:

| Tag Key | Tag Value | 설명 |
|---------|-----------|------|
| `Project` | `AWS-Lab` | 프로젝트 식별자 (고정값) |
| `Week` | `{주차}-{세션}` | 주차 및 세션 번호 (예: `5-3`, `10-1`) |
| `CreatedBy` | `Student` | 생성자 구분 (고정값) |

### 태그 추가 작성 방법

#### 표 형식 (권장)

```markdown
9. **Tags - optional** 섹션에서 [[Add new tag]] 버튼을 클릭한 후 다음 태그를 추가합니다:

| Key | Value |
|-----|-------|
| `Project` | `AWS-Lab` |
| `Week` | `5-3` |
| `CreatedBy` | `Student` |

10. [[Create table]] 버튼을 클릭합니다.
```

#### 간결 버전 (허용)

```markdown
8. **Tags - optional**에서 태그를 추가합니다: `Project=AWS-Lab`, `Week=10-1`, `CreatedBy=Student`
```

### 리소스 정리 섹션 작성

모든 실습 가이드에 다음 3가지 방법을 제공합니다:

```markdown
## 리소스 정리

> [!WARNING]
> 다음 단계를 **반드시 수행**하여 불필요한 비용을 방지하세요.

### 방법 1: Tag Editor로 리소스 찾기 (권장)

1. AWS Management Console에 로그인한 후 상단 검색창에서 `Resource Groups & Tag Editor`를 검색하고 선택합니다.
2. 왼쪽 메뉴에서 **Tag Editor**를 선택합니다.
3. **Regions**에서 `ap-northeast-2`를 선택합니다.
4. **Resource types**에서 `All supported resource types`를 선택합니다.
5. **Tags** 섹션에서 다음을 입력합니다:
   - **Tag key**: `Week`
   - **Tag value**: `{주차}-{세션}`
6. [[Search resources]] 버튼을 클릭합니다.
7. 이 실습에서 생성한 모든 리소스가 표시됩니다.

> [!NOTE]
> Tag Editor는 리소스를 찾는 용도로만 사용됩니다. 실제 삭제는 각 서비스 콘솔에서 수행해야 합니다.

### 방법 2: CloudFormation 스택 삭제 (태스크 0이 있는 경우)

1. CloudFormation 콘솔로 이동합니다.
2. `week{주차}-{세션}-lab-stack` 스택을 선택합니다.
3. [[Delete]] 버튼을 클릭합니다.
4. 확인 창에서 [[Delete]] 버튼을 클릭합니다.
5. 스택 삭제가 완료될 때까지 기다립니다 (2-3분 소요).

> [!NOTE]
> CloudFormation 스택을 삭제하면 태스크 0에서 생성한 모든 리소스가 자동으로 삭제됩니다.

### 방법 3: 수동 삭제

[각 리소스를 개별적으로 삭제하는 단계]

✅ **실습 종료**: 모든 리소스가 정리되었습니다.
```

### CloudFormation 스택 태그 자동 전파

CloudFormation 스택에 태그를 추가하면 스택이 생성하는 모든 리소스에 자동으로 전파됩니다.

**작성 예시**:
```markdown
9. **Tags - optional** 섹션에서 [[Add new tag]] 버튼을 클릭한 후 다음 태그를 추가합니다:

| Key | Value |
|-----|-------|
| `Project` | `AWS-Lab` |
| `Week` | `6-2` |
| `CreatedBy` | `Student` |

> [!NOTE]
> 이 태그들은 CloudFormation 스택이 생성하는 모든 리소스에 자동으로 전파됩니다.

10. [[Create stack]] 버튼을 클릭합니다.
```
