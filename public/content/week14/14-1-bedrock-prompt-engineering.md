> [!IMPORTANT]
> 이 실습을 시작하기 전에 AWS 리전을 **US East (N. Virginia) us-east-1**로 설정합니다.
> Claude 3 모델은 특정 리전에서만 사용 가능하며, us-east-1과 us-west-2에서 가장 안정적으로 제공됩니다.
> 이 가이드는 us-east-1 기준으로 작성되었습니다.

이 실습에서는 Amazon Bedrock Playground를 사용하여 프롬프트 엔지니어링의 핵심 기법을 학습합니다.
효과적인 프롬프트 작성 방법을 익히고, 다양한 프롬프팅 기법을 실습하여 생성형 AI의 응답 품질을 최적화하는 방법을 배웁니다.

> [!NOTE]
> Amazon Bedrock에서는 모든 Foundation Model에 대한 액세스가 기본적으로 활성화되어 있습니다.
> AWS Marketplace 권한만 있으면 바로 모델을 사용할 수 있으며, Anthropic 모델의 경우 첫 사용 시 사용 사례 양식만 제출하면 됩니다.

## 태스크 1: Playground에서 기본 프롬프트 테스트

이 태스크에서는 Amazon Bedrock Playground를 사용하여 기본적인 프롬프트를 테스트합니다.
Playground는 코드 작성 없이 브라우저에서 직접 Foundation Model과 상호작용할 수 있는 환경을 제공합니다.

### 태스크 1.1: Playground 접속 및 모델 선택

1. AWS Management Console에 로그인한 후 상단 검색창에서 `Amazon Bedrock`을 검색하고 선택합니다.
2. 왼쪽 메뉴에서 **Playgrounds** > **Text**를 선택합니다.
3. 오른쪽 패널에서 [[Select model]] 버튼을 클릭합니다.
4. `Anthropic Claude 3 Haiku` (또는 사용 가능한 최신 경량 모델)를 선택한 후 [[Apply]] 버튼을 클릭합니다.

> [!NOTE]
> Claude 3 Haiku는 빠른 응답 속도와 낮은 비용으로 프롬프트 엔지니어링 실습에 적합합니다.
> Claude 3 Sonnet은 더 높은 품질의 응답을 제공하지만 비용이 더 높습니다.
>
> 콘솔에서 Claude 3.5 모델이 표시되면 해당 모델을 사용해도 무방합니다.

✅ **하위 태스크 완료**: Playground가 준비되었습니다.

### 태스크 1.2: 기본 프롬프트 테스트

1. 채팅 입력창에 다음 프롬프트를 입력합니다: `AWS Lambda란 무엇인가요?`
2. [[Run]] 버튼(또는 전송 버튼)을 클릭합니다.

> [!NOTE]
> AWS Console UI는 시점에 따라 변경될 수 있습니다. "Run" 버튼이 화살표(▶) 아이콘으로 표시될 수 있습니다.

3. 응답을 확인합니다.

> [!OUTPUT]
>
> ```text
> AWS Lambda는 서버를 프로비저닝하거나 관리하지 않고도 코드를 실행할 수 있는
> 서버리스 컴퓨팅 서비스입니다. 이벤트에 응답하여 자동으로 코드를 실행하며,
> 사용한 컴퓨팅 시간에 대해서만 비용을 지불합니다.
> ```

> [!NOTE]
> 생성형 AI의 특성상 동일한 프롬프트에도 매번 다른 응답이 생성될 수 있습니다.
> 위 출력은 응답 예시이며, 실제 응답은 다를 수 있습니다.

4. 오른쪽 패널에서 **Token count**를 확인합니다.

> [!NOTE]
> Token count는 입력(Input tokens)과 출력(Output tokens)으로 구분됩니다.
> 토큰은 비용 계산의 기준이 되며, 한국어는 영어보다 더 많은 토큰을 사용합니다.

✅ **태스크 완료**: 기본 프롬프트 테스트가 완료되었습니다.

## 태스크 2: 프롬프트 엔지니어링 기법 실습

이 태스크에서는 프롬프트 엔지니어링의 핵심 기법인 Zero-shot, Few-shot, Chain-of-Thought 프롬프팅을 실습합니다.
각 기법의 특징과 효과를 비교하여 상황에 맞는 최적의 프롬프팅 방법을 학습합니다.

### 태스크 2.1: Zero-shot 프롬프팅

Zero-shot 프롬프팅은 예시 없이 작업을 직접 요청하는 가장 기본적인 방법입니다.

1. [[New chat]] 버튼을 클릭하여 새 대화를 시작합니다.
2. 다음 프롬프트를 입력합니다:

```text
다음 문장의 감정을 분석합니다:
"이 제품은 정말 실망스럽습니다. 기대했던 것과 완전히 다릅니다."
```

3. [[Run]] 버튼(또는 전송 버튼)을 클릭합니다.
4. 응답을 확인합니다.

> [!OUTPUT]
>
> ```text
> 이 문장은 부정적인 감정을 표현하고 있습니다.
> - 감정: 부정적 (Negative)
> - 주요 키워드: "실망스럽습니다", "완전히 다릅니다"
> - 감정 강도: 강함
> ```

> [!NOTE]
> 응답 예시입니다. 실제 응답은 다를 수 있습니다.

✅ **하위 태스크 완료**: Zero-shot 프롬프팅이 완료되었습니다.

### 태스크 2.2: Few-shot 프롬프팅

Few-shot 프롬프팅은 몇 가지 예시를 제공하여 모델이 패턴을 학습하도록 하는 방법입니다.

1. [[New chat]] 버튼을 클릭합니다.
2. 다음 프롬프트를 입력합니다:

```text
다음 예시를 참고하여 문장의 감정을 분석합니다:

예시 1:
문장: "이 제품은 정말 훌륭합니다!"
감정: 긍정적 (Positive)
점수: 9/10

예시 2:
문장: "배송이 너무 느립니다."
감정: 부정적 (Negative)
점수: 3/10

이제 다음 문장을 분석합니다:
"가격 대비 성능이 괜찮은 편입니다."
```

3. [[Run]] 버튼을 클릭합니다.
4. 응답을 확인합니다.

> [!OUTPUT]
>
> ```text
> 문장: "가격 대비 성능이 괜찮은 편입니다."
> 감정: 긍정적 (Positive)
> 점수: 5-6/10
>
> 이 문장은 제품에 대한 긍정적인 평가를 담고 있지만,
> "괜찮은 편"이라는 표현이 강한 긍정보다는 중립에 가까운 긍정을 나타냅니다.
> ```

> [!NOTE]
> 응답 예시입니다. 실제 응답은 다를 수 있습니다.
>
> Few-shot 프롬프팅은 Zero-shot보다 더 일관된 형식의 응답을 생성합니다.
> 예시를 통해 모델이 원하는 출력 형식을 학습하기 때문입니다.

✅ **하위 태스크 완료**: Few-shot 프롬프팅이 완료되었습니다.

### 태스크 2.3: Chain-of-Thought 프롬프팅

Chain-of-Thought 프롬프팅은 모델이 단계별로 사고 과정을 설명하도록 유도하는 방법입니다.

1. [[New chat]] 버튼을 클릭합니다.
2. 다음 프롬프트를 입력합니다:

```text
다음 문제를 단계별로 풀어줍니다:

문제: 한 회사가 AWS Lambda를 사용하여 월 1,000만 건의 요청을 처리합니다.
각 요청은 평균 200ms 동안 실행되며, 메모리는 512MB를 사용합니다.
월 비용은 얼마인가요?

(AWS Lambda 가격: 요청 100만 건당 $0.20, GB-초당 $0.0000166667)

단계별로 계산 과정을 보여줍니다.
```

3. [[Run]] 버튼을 클릭합니다.
4. 응답을 확인합니다.

> [!OUTPUT]
>
> ```text
> 단계별 계산:
>
> 1단계: 요청 비용 계산
> - 월 요청 수: 10,000,000건
> - 프리 티어 제외: 10,000,000 - 1,000,000 = 9,000,000건
> - 요청 비용: (9,000,000 / 1,000,000) × $0.20 = $1.80
>
> 2단계: 컴퓨팅 시간 계산
> - 총 실행 시간: 10,000,000 × 0.2초 = 2,000,000초
> - 메모리: 512MB = 0.5GB
> - GB-초: 2,000,000 × 0.5 = 1,000,000 GB-초
>
> 3단계: 컴퓨팅 비용 계산
> - 컴퓨팅 비용: 1,000,000 × $0.0000166667 = $16.67
>
> 4단계: 총 비용
> - 총 비용: $2.00 + $16.67 = $18.67
>
> 답: 월 비용은 약 $18.67입니다.
> ```

> [!NOTE]
> 응답 예시입니다. 실제 응답은 다를 수 있습니다.
>
> 이 계산은 AWS Lambda 무료 티어(월 100만 건 요청, 400,000 GB-초)를 제외한 가격 기준입니다.
>
> Chain-of-Thought 프롬프팅은 복잡한 문제 해결에 효과적입니다.
> "단계별로", "차근차근", "먼저 ~, 그 다음 ~" 같은 표현을 사용하면 더 상세한 설명을 얻을 수 있습니다.

✅ **태스크 완료**: 프롬프트 엔지니어링 기법 실습이 완료되었습니다.

## 태스크 3: 시스템 프롬프트와 파라미터 조정

이 태스크에서는 시스템 프롬프트와 모델 파라미터를 조정하여 응답 품질을 최적화하는 방법을 학습합니다.
시스템 프롬프트는 모델의 역할과 행동을 정의하고, 파라미터는 응답의 창의성과 일관성을 제어합니다.

### 태스크 3.1: 시스템 프롬프트 설정

1. [[New chat]] 버튼을 클릭합니다.
2. 오른쪽 패널에서 **System prompts** 섹션을 찾습니다.

> [!NOTE]
> AWS Console UI는 시점에 따라 변경될 수 있습니다.
> "Add system prompt" 버튼이 별도로 표시되지 않고, 시스템 프롬프트 입력 영역이 바로 노출될 수 있습니다.

3. 시스템 프롬프트 입력 영역에 다음을 입력합니다:

```text
당신은 AWS 클라우드 아키텍처 전문가입니다.
사용자의 질문에 대해 다음 형식으로 답변합니다:

1. 핵심 개념 설명.
2. AWS 서비스 추천.
3. 모범 사례.
4. 주의사항.

답변은 간결하고 실용적이어야 하며, 기술 용어는 쉽게 설명합니다.
```

> [!NOTE]
> 시스템 프롬프트를 입력하면 자동으로 적용됩니다. 별도의 저장 버튼이 없습니다.

4. 채팅 입력창에 다음 질문을 입력합니다: `서버리스 아키텍처를 구축하려면 어떻게 해야 하나요?`.
5. [[Run]] 버튼을 클릭합니다.
6. 응답을 확인합니다.

> [!OUTPUT]
>
> ```text
> 1. 핵심 개념 설명
> 서버리스 아키텍처는 서버 관리 없이 애플리케이션을 실행하는 방식입니다.
> 인프라 관리는 클라우드 제공자가 담당하고, 개발자는 코드 작성에만 집중합니다.
>
> 2. AWS 서비스 추천
> - AWS Lambda: 이벤트 기반 코드 실행
> - Amazon API Gateway: RESTful API 생성
> - Amazon DynamoDB: NoSQL 데이터베이스
> - Amazon S3: 정적 콘텐츠 호스팅
>
> 3. 모범 사례
> - 함수는 단일 책임 원칙을 따르도록 작게 유지
> - 환경 변수로 설정 관리
> - Amazon CloudWatch로 모니터링 및 로깅
>
> 4. 주의사항
> - 콜드 스타트 지연 시간 고려
> - 동시 실행 제한 확인
> - 비용 최적화를 위한 메모리 및 타임아웃 조정
> ```

> [!NOTE]
> 응답 예시입니다. 실제 응답은 다를 수 있습니다.
>
> 시스템 프롬프트는 모든 대화에 적용되는 기본 지침입니다.
> 모델의 역할, 응답 형식, 톤앤매너를 일관되게 유지할 수 있습니다.

✅ **하위 태스크 완료**: 시스템 프롬프트가 설정되었습니다.

### 태스크 3.2: Temperature 파라미터 조정

Temperature는 응답의 창의성과 무작위성을 제어하는 파라미터입니다.

1. [[New chat]] 버튼을 클릭합니다.

> [!NOTE]
> 이전 태스크에서 설정한 시스템 프롬프트는 New chat을 클릭해도 유지됩니다.
> Temperature 비교 실험에서는 시스템 프롬프트를 제거하고 진행하는 것을 권장합니다.
> 시스템 프롬프트 입력 영역을 비워서 제거할 수 있습니다.

2. 오른쪽 패널에서 **Inference configurations** 섹션을 찾습니다.

> [!NOTE]
> AWS Console UI는 시점에 따라 변경될 수 있습니다.
> "Inference configurations" 대신 "Inference parameters"로 표시될 수 있습니다.

3. **Temperature** 슬라이더를 `0.1`로 설정합니다.

Temperature 값의 의미를 이해합니다.

> [!NOTE]
> Temperature 값의 의미:
>
> - 0에 가까울수록: 일관되고 예측 가능한 응답 (사실 기반 질문에 적합)
> - 1에 가까울수록: 창의적이고 다양한 응답 (창작 작업에 적합)

4. 채팅 입력창에 다음을 입력합니다: `AWS Lambda의 장점 3가지를 알려주세요.`.
5. [[Run]] 버튼을 클릭합니다.
6. 응답을 확인하고 메모장에 복사합니다.
7. [[New chat]] 버튼을 클릭합니다.
8. **Temperature** 슬라이더를 `0.9`로 설정합니다.
9. 동일한 질문을 입력합니다: `AWS Lambda의 장점 3가지를 알려주세요.`.
10. [[Run]] 버튼을 클릭합니다.
11. 응답을 확인하고 이전 응답과 비교합니다.

두 응답을 비교하여 다음을 확인합니다:

- **표현 방식**: Temperature 0.1은 정형화된 표현, 0.9는 다양한 표현 사용
- **내용 구성**: 0.1은 일반적인 답변, 0.9는 독창적인 관점 포함 가능
- **일관성**: 동일 질문을 반복 실행했을 때 0.1은 유사한 답변, 0.9는 매번 다른 답변

> [!NOTE]
> Temperature 0.1에서는 일관된 표준 답변을 제공하고,
> Temperature 0.9에서는 더 다양하고 창의적인 표현을 사용합니다.

✅ **하위 태스크 완료**: Temperature 파라미터 조정이 완료되었습니다.

### 태스크 3.3: Top P 파라미터 조정

Top P는 응답 생성 시 고려할 토큰의 범위를 제어하는 파라미터입니다.

1. [[New chat]] 버튼을 클릭합니다.
2. **Inference configurations** 섹션에서 **Top P** 슬라이더를 `0.5`로 설정합니다.

> [!NOTE]
> "Inference configurations" 대신 "Inference parameters"로 표시될 수 있습니다.

Top P 값의 의미를 이해합니다.

> [!NOTE]
> Top P 값의 의미:
>
> - 낮은 값 (0.1-0.5): 가장 확률이 높은 토큰만 사용 (일관성 높음)
> - 높은 값 (0.9-1.0): 더 많은 토큰 후보 고려 (다양성 높음)
>
> Temperature와 Top P는 함께 사용하여 응답의 품질을 조정합니다.

3. 채팅 입력창에 다음을 입력합니다: `서버리스 아키텍처의 단점을 설명해주세요.`.
4. [[Run]] 버튼을 클릭합니다.
5. 응답을 확인합니다.

✅ **태스크 완료**: 시스템 프롬프트와 파라미터 조정이 완료되었습니다.

## 📚 참고: 프롬프트 엔지니어링 심화

### 프롬프트 엔지니어링이란

**정의:**
프롬프트 엔지니어링은 생성형 AI 모델로부터 원하는 응답을 얻기 위해 입력(프롬프트)을 최적화하는 기술입니다.
효과적인 프롬프트는 모델의 성능을 극대화하고 비용을 절감할 수 있습니다.

**중요성:**

- 동일한 모델에서도 프롬프트에 따라 응답 품질이 크게 달라집니다.
- 잘 설계된 프롬프트는 모델 파인튜닝 없이도 높은 성능을 발휘합니다.
- 토큰 사용을 최적화하여 비용을 절감할 수 있습니다.

### 프롬프팅 기법 비교

**Zero-shot 프롬프팅:**

- 예시 없이 작업을 직접 요청합니다.
- 장점: 간단하고 빠름, 토큰 사용 최소화합니다.
- 단점: 복잡한 작업에서 일관성이 부족합니다.
- 사용 사례: 간단한 질문, 일반적인 작업입니다.

**Few-shot 프롬프팅:**

- 몇 가지 예시를 제공하여 패턴을 학습합니다.
- 장점: 일관된 형식, 높은 정확도를 제공합니다.
- 단점: 토큰 사용이 증가하고, 예시 작성이 필요합니다.
- 사용 사례: 특정 형식의 출력, 분류 작업입니다.

**Chain-of-Thought 프롬프팅:**

- 단계별 사고 과정을 유도합니다.
- 장점: 복잡한 문제를 해결하고, 추론 과정을 확인할 수 있습니다.
- 단점: 응답 시간이 증가하고, 토큰 사용이 많습니다.
- 사용 사례: 수학 문제, 논리적 추론, 복잡한 분석입니다.

### 시스템 프롬프트 설계 원칙

**명확한 역할 정의:**

```
당신은 AWS 솔루션 아키텍트입니다.
고객의 요구사항을 분석하고 최적의 AWS 아키텍처를 제안합니다.
```

**응답 형식 지정:**

```
모든 답변은 다음 형식을 따르세요:
1. 요약 (2-3문장)
2. 상세 설명
3. 코드 예시 (필요시)
4. 참고 자료
```

**제약 조건 명시:**

```
- 답변은 500자 이내로 작성
- 기술 용어는 쉽게 설명
- 코드는 Python으로 작성
```

**톤앤매너 설정:**

```
친근하고 전문적인 톤으로 답변하세요.
전문 용어는 사용하되, 초보자도 이해할 수 있도록 설명을 추가하세요.
```

### 모델 파라미터 가이드

**Temperature (0.0 - 1.0):**

- **0.0 - 0.3**: 사실 기반 질문, 코드 생성, 번역
  - 예: "AWS Lambda의 가격 정책은?"
- **0.4 - 0.7**: 일반적인 대화, 요약, 분석
  - 예: "이 아키텍처의 장단점을 분석해주세요"
- **0.8 - 1.0**: 창작, 브레인스토밍, 마케팅 문구
  - 예: "서버리스 제품의 캐치프레이즈를 만들어주세요"

**Top P (0.0 - 1.0):**

- **0.1 - 0.5**: 일관성이 중요한 작업
  - 예: 기술 문서 작성, 정책 설명
- **0.6 - 0.8**: 균형잡힌 응답
  - 예: 일반적인 질의응답
- **0.9 - 1.0**: 다양성이 중요한 작업
  - 예: 아이디어 생성, 창작

> [!NOTE]
> **Temperature와 Top P 상호작용:**
> Temperature와 Top P는 일반적으로 하나만 조정하는 것을 권장합니다.
> 둘 다 높게 설정하면 응답이 지나치게 무작위해질 수 있습니다.
> Anthropic은 Temperature 또는 Top P 중 하나만 변경하고 나머지는 기본값으로 유지할 것을 권장합니다.

**Max Tokens:**

- 응답의 최대 길이 제어
- 비용 관리 및 응답 시간 최적화
- Claude 3 Haiku/Sonnet: 최대 4,096 토큰
- Claude 3.5 Haiku/Sonnet: 최대 8,192 토큰
- 권장: 짧은 답변 500, 중간 1000, 긴 답변 2000
- 사용 중인 모델의 공식 문서에서 최대 토큰 수를 확인하세요

### 효과적인 프롬프트 작성 팁

**구체적으로 작성:**

```
❌ "AWS에 대해 설명해주세요"
✅ "AWS Lambda의 콜드 스타트 문제와 해결 방법을 3가지 알려주세요"
```

**컨텍스트 제공:**

```
❌ "이 코드를 최적화해주세요"
✅ "다음은 Python으로 작성된 AWS Lambda 함수입니다.
    메모리 사용을 줄이고 실행 시간을 개선해주세요:
    [코드]"
```

**출력 형식 명시:**

```
❌ "AWS 서비스를 비교해주세요"
✅ "Amazon S3와 EBS를 다음 형식의 표로 비교해주세요:
    | 항목 | Amazon S3 | EBS |
    | 용도 | ... | ... |"
```

**단계별 지시:**

```
❌ "아키텍처를 설계해주세요"
✅ "다음 단계로 아키텍처를 설계해주세요:
    1. 요구사항 분석
    2. 적합한 AWS 서비스 선택
    3. 아키텍처 다이어그램 설명
    4. 비용 추정"
```

### 프롬프트 최적화 전략

**반복적 개선:**

1. 초기 프롬프트 작성
2. 응답 확인 및 문제점 파악
3. 프롬프트 수정 (구체화, 제약 추가)
4. 재테스트 및 비교
5. 최적 프롬프트 선택

**A/B 테스트:**

- 동일한 질문에 대해 여러 프롬프트 버전 테스트
- Temperature와 Top P 조합 실험
- 응답 품질, 속도, 비용 비교

**프롬프트 템플릿 구축:**

```
[역할 정의]
당신은 {전문 분야} 전문가입니다.

[작업 설명]
{구체적인 작업 내용}

[제약 조건]
- {제약 1}
- {제약 2}

[출력 형식]
{원하는 형식}
```

> [!NOTE]
> 중괄호 `{}`는 변수 치환을 나타냅니다. 실제 프롬프트 작성 시 중괄호 안의 내용을 실제 값으로 대체하세요.
> 예: `{전문 분야}` → `AWS 솔루션 아키텍트`

### 실무 활용 사례

**고객 지원 챗봇:**

```
시스템 프롬프트:
당신은 AWS 기술 지원 전문가입니다.
고객의 문제를 친절하게 해결하고, 단계별 가이드를 제공합니다.
답변은 다음 형식을 따르세요:
1. 문제 확인
2. 해결 방법 (단계별)
3. 추가 도움말

Temperature: 0.3 (일관된 답변)
Top P: 0.5 (정확성 우선)
```

**코드 리뷰 자동화:**

```
시스템 프롬프트:
당신은 시니어 개발자입니다.
코드를 리뷰하고 다음 항목을 확인합니다:
- 보안 취약점
- 성능 최적화 기회
- 코드 가독성
- 모범 사례 준수

Temperature: 0.2 (객관적 평가)
Top P: 0.4 (일관된 기준)
```

**마케팅 콘텐츠 생성:**

```
시스템 프롬프트:
당신은 크리에이티브 카피라이터입니다.
제품의 특징을 강조하는 매력적인 문구를 작성합니다.
톤: 친근하고 열정적
길이: 50-100자

Temperature: 0.8 (창의적)
Top P: 0.9 (다양한 표현)
```

### 비용 최적화

**토큰 사용 최소화:**

- 불필요한 예시 제거
- 간결한 시스템 프롬프트 작성
- 응답 길이 제한 (Max Tokens)

**적절한 모델 선택:**

- 간단한 작업: Claude 3 Haiku (저비용)
- 복잡한 작업: Claude 3 Sonnet (고품질)
- 최고 품질 필요: Claude 3.5 Sonnet (콘솔에서 사용 가능한 최신 고성능 모델)

> [!NOTE]
> 이 실습에서는 Claude 3 Haiku를 사용하지만, 실무에서는 작업 복잡도에 따라 모델을 선택합니다.
> Claude 3.5 Sonnet은 가장 높은 품질을 제공하지만 비용이 높으므로, 중요한 작업에 사용하는 것을 권장합니다.
> 콘솔에서 사용 가능한 최신 모델을 확인하여 선택하세요.

**캐싱 활용:**

- 반복되는 질문은 캐싱하여 재사용
- 시스템 프롬프트는 한 번만 전송
- 대화 컨텍스트 최소화

### 프롬프트 엔지니어링 체크리스트

- [ ] 필요한 출력 형식을 정의했는가?
- [ ] 제약 조건을 파악했는가?

- [ ] 일관성이 있는가?

- [ ] 개선할 부분이 있는가?
